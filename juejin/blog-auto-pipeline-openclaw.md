如果你写完一篇文章，然后立刻失去了继续做下面这些事情的欲望：

- 翻译它，
- 挪文件，
- 生成配图，
- 发到各个平台，

……恭喜，你是个正常人。

这篇文章介绍了我用 **OpenClaw** 搭建的一条小型自动化流水线：只写一次（用英文），然后让定时任务去处理那些无聊的部分——**图片生成**、**Zenn 发布（日文）**、以及**掘金预发布（中文）**。

## 1) 我们在解决什么问题？

写作本身是有趣的。"运营式博客"不是。

烦人的事情通常出现在草稿完成之后：

- "这里应该加个图……但是好懒。"
- "这篇应该翻译一下……以后再说。"（旁白：永远不会再说。）
- "我发没发到 Zenn？掘金那边准备好了吗？"

所以目标很简单：

> **创作的事情手动做，重复的事情自动化。**

---

## 2) 流水线整体结构

你在 Hugo 里写一篇英文文章（`content/posts/*.md`）。文章中可以留下**图片占位符**，标记出需要示意图或插图的位置。

然后自动化开始接管：

1. **Post Finalizer**（早上 8:00）
   - 找到包含图片占位符的新文章
   - 用 AI 图像模型生成图片
   - 把真实的图片链接插入 markdown
2. **Zenn Publisher**（早上 9:00）
   - 将文章翻译成日文
   - 通过仓库自动化发布到 Zenn
3. **Juejin Publisher**（早上 9:00）
   - 将文章翻译成简体中文
   - 把准备好的 markdown 文件存入仓库（手动发布）

![一条简单的三阶段流水线图（写作 → 图片最终化 → 分发到 Zenn/掘金），附带 8:00 和 9:00 的时钟小图标](/images/blog-auto-pipeline-openclaw/image-1.png)

核心思路：**英文原文是唯一的真相来源**，其他所有版本都从它派生。

---

## 3) "最终化"是什么意思

"最终化"意味着：

- 文章内容写得差不多了，并且
- 所有图片占位符都已被替换成真实图片。

在草稿里，我会这样添加图片注释：

（注：占位符格式为 `<!-- IMAGE: 描述 -->` + `<!-- PROMPT: 生成提示词 -->`）

Post Finalizer 任务会：

- 在 `static/images/<slug>/image-N.png` 生成 **1K 分辨率**的图片，并
- 将占位符块替换为：

```md
![描述](/images/<slug>/image-N.png)
```

这样图片路径统一，和 Hugo 的约定完全兼容。

---

## 4) Zenn + 掘金分发

图片就位之后，翻译就轻松多了，因为：

- 文章结构已经稳定，
- 图片引用已经写好，
- 翻译器只需要做该做的事：

- 保持代码块和文件名不变，
- 把正文翻译得自然流畅，
- 保持文章的可读性。

Zenn 拿到日文发布版本。掘金拿到中文 markdown 文件——提交到仓库，等待手动上传。

---

## 5) 踩坑记录（也就是现实）

测试阶段发现了几个需要注意的点：

- **时序问题：** 图片生成比较慢，所以 Finalizer 要早于翻译任务运行。
- **幂等性：** 每个任务都用一个 JSON 文件记录状态，不会重复执行已完成的工作。
- **跨平台图片：** Hugo 喜欢 `static/` 目录；其他平台有自己的规则，需要单独处理。

## 6) 小结

这条流水线的目标不是"把写作也自动化"，而是做一件更有用的事：

- **保护写作时间**，同时
- **消灭写完之后的杂务**，因为正是这些杂务让人慢慢放弃更新。

如果你也想搭一套类似的东西，从两条规则开始：

1. 把你的风格规范集中在一个地方（我用的是 `CLAUDE.md`）。
2. 把所有"复制粘贴加几步操作"的事情自动化掉。

好了，让我把剩下的事情交给 cron——毕竟我最高效的生产力技巧，还是：*在自动化跑任务的时候睡觉。*
