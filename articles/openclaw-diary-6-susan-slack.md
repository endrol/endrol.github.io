## 1. Susanとは何者か？

**Susan** — OpenClawの2人目のエージェントで、ブログ関連のことは実質彼女が担当している。

役割はブログの執筆と、投稿に関わるcronジョブのすべて。下書き、最終化、翻訳、公開——パイプライン全体だ。ブログ記事が存在している（あるいは新たに生まれる）なら、それはSusanの仕事だ。

1人目のエージェント **Ada** は残りすべてを担当する。Adaを汎用ブレイン、Susanをコンテンツフォルダに住む専門家と思えばわかりやすい。

名前と役割を明確に分けることで、何が起きているか判断しやすくなる。Slackで @ するときも迷わなくて済む。

## 2. Slackのセットアップ（地味なパート）

@ メンションやエージェントとの会話を始める前に、まずSlackを繋げる作業が必要だ。それをやったのは私で、思ったより時間がかかった。

全体の手順をまとめると：

**ステップ1 — ワークスペースとSlackアプリの作成**

まずSlackワークスペースが必要だ。用意できたら [api.slack.com/apps](https://api.slack.com/apps) にアクセスし、新しいアプリを作成する。「ゼロから作成」を選んで、名前とワークスペースを指定するだけ。ここは簡単。

**ステップ2 — OAuthとパーミッションの設定（ボットユーザートークン）**

多くのチュートリアルが人を迷子にするのはここだ。サイドバーの **OAuth & Permissions** に移動し、**Bot Token Scopes** までスクロールして、実際に必要なパーミッションを追加する。`chat:write`、`app_mentions:read`、`channels:history` など。

スコープを設定したらアプリをワークスペースにインストールする。Slackから **Bot User OAuth Token** が発行される——`xoxb-` から始まるものだ。これがボットがメッセージを投稿したりチャンネルを読んだりするためのトークン。

**ステップ3 — アプリレベルトークンの取得**

ボットトークンだけでは、Socket Modeを使う場合には不十分だ。Socket Modeを使うと、公開HTTPエンドポイントなしにイベントを受け取れる——ローカルやセルフホスト環境には非常に便利な機能だ。これには **アプリレベルトークン** が必要になる。

**Basic Information → App-Level Tokens** でトークンを生成し、`connections:write` スコープを付与する。`xapp-` から始まるトークンが手に入る。両方のトークンは安全な場所に保管しておこう。

**ステップ4 — Socket Modeを有効化**

**Socket Mode** でトグルをオンにする。これでSlackがWebSocket接続を通じてイベントを届けてくれるようになり、webhookのURLは不要になる。

**ステップ5 — イベントのサブスクライブ**

最後に **Event Subscriptions** でイベントを有効化し、必要なものをサブスクライブする。OpenClawの場合は `app_mention` ——エージェントが実際に @ タグされたときだけ起動するようにするためだ。

以上が完了すると、2つのトークンと、リスン・レスポンスができるボットが手に入る。OAuthのセットアップで頭を抱えないよう、[このYouTubeチュートリアル](https://www.youtube.com/watch?v=9QpSkGnfKMk)を参考にした——Slackのダッシュボードが迷路に見えてきたら一見の価値あり。

## 3. Slackでのエージェントとの会話

現在のインタラクションモデルはシンプルだ：**私が @ すると、エージェントが返答する**。

```
@Susan — openclaw diary 6の投稿を最終化して
@Ada   — パイプラインの状況は？
```

エージェントはSlackチャンネルをリッスンし、メンションを拾い、処理して、返信する。十分にシンプルだ。

OpenClawに組み込まれた重要なルールがある：**ボットは他のボットからのメッセージを無視する**。SusanはAdaが何か言っても反応しない。AdaもSusanには反応しない。彼女たちが反応するのは私だけだ。

## 4. ボット同士の問題

これは手抜きではなく、意図的な設計だ。理由は単純：**無限ループ**。

何も制御しなければ、お互いのメッセージに返信し合う2つのボットは、永遠に続いて何も生産しない会話に突入する。互いに @ し合う2つのボットは、向かい合った2枚の鏡と同じ——技術的には無限で、完全に意味がない。

![2体のロボットが互いに@シンボルを飛ばし合い、無限ループする](/images/openclaw-diary-6-susan-slack/image-2.png)

OpenClawはチャンネルごとの `allowBots` 設定でこれを解決している。デフォルトは `false` だ：

```json
{
  "channels": {
    "slack": {
      "channels": {
        "C0AKYTMK1QR": {
          "allowBots": true
        }
      }
    }
  }
}
```

`true` にするとそのチャンネルでボット同士がメッセージを読めるようになる。ただし、ループ検出やメッセージの重複排除といった追加の仕組みがなければ、これは地雷だ。だから今は `false` のままにしている。

## 5. 本当にやりたいこと：能動的なトリガー

現在の仕組みは **リアクティブ** だ：私がエージェントに何かを指示し、エージェントが実行する。

最終的にやりたいのは **プロアクティブ** な動き：Adaが投稿の準備ができたことを検知し、Susanに通知し、Susanがパイプラインを開始する——人間が介在せずに。

これが自然な次のステップだ。状態の変化によってトリガーされる、エージェント間のハンドオフ。こんなイメージ：

```
Ada:   「Susanへ、新しい投稿を検出。最終化する？」
Susan: 「了解、やります。」
```

ただし、これには `allowBots` を切り替える以上の仕組みが必要だ。適切なハンドオフロジック、ループ防止、そしてAdaが渡すものとSusanが受け取るものの明確な契約が必要になる。やる価値はある——でも今日じゃない。

## まとめ

2人のエージェントは、それぞれに明確な役割があって、互いに意図せず会話し合わない限り、1人より優れている。

SusanはブログをAda はそれ以外を担当する。必要なほうに @ する。`allowBots` の壁が今のところ正気を保ってくれている。

次は、私を介さずに2人が実際に連携できるようにすること。そこが面白いところで——次の日記のネタにもなる。

乞うご期待。Susanがすでに書き始めている。
